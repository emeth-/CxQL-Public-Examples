General information about rules

First, let's look at a few basic concepts and the process of working with rules, for a better understanding of what will happen next. And also because it is not mentioned in the documentation or is spread out over the structure, which is not very convenient.

  1. Rules are applied during scanning depending on the preset selected at the start (set of active rules). You can create an unlimited number of presets and how exactly to structure them depends on the features of your process. You can group them by languages or allocate presets for each project. The number of active rules affects the speed and accuracy of scanning.

  2. Rules are edited in a special tool called CxAuditor. This is a desktop application that connects to the Checkmarx server. This tool has two modes of operation: editing rules and analyzing the results of an already conducted scan.

  3. Rules in Checkmarx are divided by languages, that is, for each language there is its own set of queries. There are also some general rules that are applied regardless of the language, these are so-called base queries. Mostly, base queries contain information that other rules use.

  4. Rules can be “Executable” and “Non-Executable”. Not quite a correct name, in my opinion, but it is what it is. The essence is that the result of executing “Executable” rules will be displayed in the scan results in the UI, and “Non-Executable” rules are needed only to use their results in other queries (essentially - just a function).

  5. You can create new rules or supplement/override existing ones. To override a rule, you need to find it in the tree, right-click and select “Override” in the drop-down menu. It is important to remember that new rules are initially not included in the presets and are not active. To start using them, you need to activate them in the “Preset Manager” menu in the tool. Overridden rules retain their settings, that is, if the rule was active, it will remain so and will be applied immediately.

  6. During execution, a “tree” of queries is built, showing dependencies. First, rules that collect information are executed, followed by those that use it. The execution result is cached, so if there is an opportunity to use the results of an existing rule, it is better to do so, as it will reduce the scan time.

  7. Rules can be applied at different levels:

  • For the entire system - will be used for any scan of any project

  • At the team level - will be applied only for scanning projects in the selected team.

  • At the project level - Will be applied in a specific project

“Dictionary” for beginners

And I will start with a few things that puzzled me, and also show a number of techniques that will significantly simplify life.

List operations

- subtraction of one from the other (list2 - list1)
* intersection of lists (list1 * list2)
+ addition of lists (list1 + list2)

& (logical AND) - combines lists by matching (list1 & list2), similar to intersection (list1 * list2)
| (logical OR) - combines lists by broad search (list1 | list2)

Does not work with lists:  ^  &&  ||  %  /

All found elements

Within the scanned language, you can get a list of absolutely all elements that Checkmarx has identified (strings, functions, classes, methods, etc.). This is a certain space of objects that can be accessed through All. That is, to search for an object with a specific name searchMe, you can perform a search, for example, by name for all found objects: 

// This query will return all elements
result = All;

// This query will return all elements that contain “searchMe” in their name
result = All.FindByName("searchMe");

But, if you need to search in another language that for some reason was not included in the scan (for example, groovy in an Android project), you can expand our object space through a variable: 

result = AllMembers.All.FindByName("searchMe");

Functions for Flow analysis

These functions are used in many rules and here is a small cheat sheet on what they mean:

// What data second influences first.
// In other words - THAT (second) that influences ME (first).
result = first.DataInfluencedBy(second);

// What data first influences second.
// In other words - I (first) influence THAT (second).
result = first.DataInfluencingOn(second);

Getting file name/path

There are several attributes that can be obtained from the query execution results (the name of the file in which the occurrence was found, the line, etc.), but how to get and use them is not mentioned in the documentation. So, to do this, you need to refer to the LinePragma property, and inside it will be the objects we need: 

// For example, let's find all methods
CxList methods = Find_Methods();

// Find the method scope by name in the methods
CxList scope = methods.FindByName("scope");

// This way you can get the file path
string current_filename = scope.GetFirstGraph().LinePragma.FileName;

// And this way - the line where the hit was found
int current_line = scope.GetFirstGraph().LinePragma.Line;

// These parameters can be used in different ways
// For example, get all objects in the file
CxList inFile = All.FindByFileName(current_filename);

// Or find out what happens in a specific line
CxList inLine = inFile.FindByPosition(current_line);

It should be noted that FileName actually contains the file path, since we used the GetFirstGraph method.

Execution result

Inside CxQL, there is a special variable result that returns the result of executing your written rule. It is initialized immediately and you can write intermediate results to it, changing and refining them in the process. But, if there is no assignment to this variable or a return function inside the rule, the execution result will always be zero.

The following query will not return anything in the execution result and will always be empty:

// Find foo elements
CxList libraries = All.FindByName("foo");

But, by assigning the execution result to the magic variable result, we will see what this call returns:

// Find foo elements
CxList libraries = All.FindByName("foo");

// Output as the execution result of the rule
result = libraries

// Or even shorter
result = All.FindByName("foo");

Using the results of other rules

Rules in Checkmarx can be called an analogue of functions in a regular programming language. When writing a rule, you can quite use the results of other queries. For example, there is no need to search for all method calls in the code every time, just call the desired rule: 

// Get the result of another rule
CxList methods = Find_Methods();

// Search inside the method foo. 
// The second parameter false means that the search is case-insensitive
result = methods.FindByShortName("foo", false);

This approach allows you to reduce the code and significantly reduce the rule execution time.

Problem solving

Logging

When working with the tool, sometimes it is not possible to immediately write the desired query and you have to experiment, trying different options. For such a case, the tool provides logging, which is called as follows: 

// Find something
CxList toLog = All.FindByShortName("log");

// Form a string and send it to the log
cxLog.WriteDebugMessage (“number of DOM elements =” + All.Count);

But it is worth remembering that this method only accepts a string as input, so it will not be possible to output a complete list of found elements as a result of the first operation. The second option used for debugging is to periodically assign the result of the query execution to the magic variable result and see what happens. This approach is not very convenient, you need to be sure that there is no reassignment or operations with this result in the code below or just comment out the code located below. Or you can, like me, forget to remove a few such calls from the finished rule and wonder why nothing works.

A more convenient way is to call the return method with the desired parameter. In this case, the rule execution will end and we will be able to see what happened as a result of our writing: 

// Find something
CxList toLog = All.FindByShortName("log");

// Output the execution result
return toLog

//Everything written below will not be executed
result = All.DataInfluencedBy(toLog)

Login problem

There are situations when it is not possible to log in to the CxAudit tool (which is used for writing rules). There can be many reasons for this, such as an unexpected Windows update, BSOD, and other unforeseen situations that are beyond our control. In such a case, sometimes an unfinished session remains in the database, which prevents re-login. To fix this, you need to execute a few queries:

For Checkmarx up to 8.6:

// Check if there are logged-in users by executing a query in the database
SELECT COUNT(*) FROM [CxDB].[dbo].LoggedinUser WHERE [ClientType] = 6;
 
// If there is something, or even if there is not, try executing the query
DELETE FROM [CxDB].[dbo].LoggedinUser WHERE [ClientType] = 6;

For Checkmarx after 8.6:

// Check if there are logged-in users by executing a query in the database
SELECT COUNT(*) FROM LoggedinUser WHERE (ClientType = 'Audit');
 
// If there is something, or even if there is not, try executing the query
DELETE FROM [CxDB].[dbo].LoggedinUser WHERE (ClientType = 'Audit');

Writing rules

Finally, we have reached the most interesting part. When you start writing rules in CxQL, you often lack not so much documentation as live examples of solving specific tasks and descriptions of the query operation process as a whole.

I will try to simplify the life of those who are starting to dive into the query language and provide several examples of using Custom Queries to solve specific tasks. Some of them are quite general and can be applied in your company with little or no changes, while others are more specific, but they can also be used by adapting the code to the specifics of your applications.

So, here are the tasks we most often encountered:

Task: In the results of the rule execution, there are several Flows, and one of them is nested within the other, it is necessary to leave only one of them.

Solution: Indeed, sometimes Checkmarx shows several Flows of data movement that can intersect and be a shortened version of others. For such cases, there is a special method ReduceFlow. Depending on the parameter, it will select the shortest or the longest Flow: 

// Leave only long Flows
result = result.ReduceFlow(CxList.ReduceFlowType.ReduceSmallFlow);

// Leave only short Flows
result = result.ReduceFlow(CxList.ReduceFlowType.ReduceBigFlow);

Task: Expand the list of sensitive data that the tool reacts to

Solution: In Checkmarx, there are base rules, the results of which are used by many other queries. By supplementing some of these rules with data specific to your application, you can immediately improve the scan results. Below is an example of a rule to start with:

General_privacy_violation_list

Add a few variables that are used in our application to store sensitive information:

// Get the result of the base rule
result = base.General_privacy_violation_list();

// Search for elements that fall under simple regular expressions. You can supplement with patterns characteristic of you.
CxList personalList = All.FindByShortNames(new List<string> {
	"*securityToken*", "*sessionId*"}, false);

// Add to the final result
result.Add(personalList);

Solution: All queries in Checkmarx are divided by languages, so you need to supplement the rules for each language. Below are a few examples of such rules.

If libraries are used that supplement or replace standard functionality, they can be easily added to the base rule. Then everyone who uses it will immediately know about the new inputs. For example, logging libraries in Android - Timber and Loggi. In the base set of rules, there are no definitions of non-system calls, so if a password or session identifier gets into the log, we will not know about it. package com.death.timberdemo;

import android.support.v7.app.AppCompatActivity;

FindAndroidOutputs 

// Get the result of the base rule
result = base.Find_Android_Outputs();

// Supplement with calls coming from the Timber library
CxList timber = All.FindByExactMemberAccess("Timber.*") +
    All.FindByShortName("Timber").GetMembersOfTarget();

Task: Search for sensitive data in plist for iOS projects

Solution: Often, various variables and values are stored in iOS using special files with the .plist extension. Storing passwords, tokens, keys, and other sensitive data in these files is not recommended, as they can be easily extracted from the device.

Plist files have features that are not obvious to the naked eye but are important for Checkmarx. Let's write a rule that will search for the data we need and notify us if passwords or tokens are mentioned somewhere.

Example of such a file, which contains a token for communication with the backend service:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
	<key>DeviceDictionary</key>
	<dict>
		<key>phone</key>
		<string>iPhone 6s</string>
	</dict>
	<key>privatekey</key>
	<string>MIICXAIBAAKBgQCqGKukO1De7zhZj6+</string>
</dict>
</plist>

And a rule for Checkmarx, which has a few nuances to consider when writing:

// Use the result of the rule for finding plist elements to reduce the rule execution time and 
CxList plist = Find_Plist_Elements();

// Initialize a new variable
CxList dictionarySettings = All.NewCxList();

// Now add the search for all the values we are interested in. In the future, you can expand this list.
// To search for values, oddly enough, FindByMemberAccess is used - searching for method calls. The second parameter inside the function, false, means that the search is case-insensitive
dictionarySettings.Add(plist.FindByMemberAccess("privatekey", false));
dictionarySettings.Add(plist.FindByMemberAccess("privatetoken", false));

// For correct search due to the features of the plist structure - you need to search by the type "If statement"
CxList ifStatements = plist.FindByType(typeof(IfStmt));

// Add to the result, before that getting the parent node - for correct display
result = dictionarySettings.FindByFathers(ifStatements);

Task: Search for information in XML

Solution: Checkmarx has very convenient functions for working with XML and searching for values, tags, attributes, and more. But the documentation, unfortunately, contains an error due to which none of the examples work. Although this shortcoming has been corrected in the latest version of the documentation, be careful if you are using earlier versions of the documents.

Here is an incorrect example from the documentation:

// The code will not work
result = All.FindXmlAttributesByNameAndValue("*.app", 8, “id”, "error- section", false, true);

As a result of trying to execute it, we will get an error that All does not have such a method… And this is true, because to use functions for working with XML, there is a special, separate object space - cxXPath. Here is what a correct query for finding a setting in Android that allows the use of HTTP traffic looks like: 

// Correct option using cxXPath
result = cxXPath.FindXmlAttributesByNameAndValue("*.xml", 8, "cleartextTrafficPermitted", "true", false, true);

Let's break it down a little more, as the syntax of all functions is similar, once you understand one, you just need to choose the right one. So, sequentially by parameters:

  • "*.xml"- file mask to search

  • 8 - language id for which the rule is applied

  • "cleartextTrafficPermitted"- attribute name in xml

  • "true" - value of this attribute

  • false - use of regular expression in search

  • true - means that the search will be performed case-insensitively

For example, a rule is used that determines incorrect, from a security point of view, network connection settings in Android that allow communication with the server via the HTTP protocol. Example of a setting containing the cleartextTrafficPermitted attribute with the value true: 

<network-security-config>
    <domain-config>
        <domain includeSubdomains="true">example.com</domain>
        <trust-anchors>
            <certificates src="@raw/my_ca"/>
        </trust-anchors>
        <domain-config cleartextTrafficPermitted="true">
            <domain includeSubdomains="true">secure.example.com</domain>
        </domain-config>
    </domain-config>
</network-security-config>

Task: Limit results by file name/path

Solution: In one of the large projects related to the development of a mobile application for Android, we encountered false positives of the rule that determines the obfuscation setting. The fact is that the out-of-the-box rule searches in the build.gradle file for the setting responsible for applying obfuscation rules for the release version of the application.

But in large projects, there are sometimes child build.gradle files that relate to libraries included in the project. The feature is that even if these files do not specify the need for obfuscation, the parent build file settings will be applied during compilation.

Thus, the task is to cut off triggers in child files that relate to libraries. They can be identified by the presence of the line apply 'com.android.library'.

Example code from the build.gradle file that defines the need for obfuscation:

apply plugin: 'com.android.application'

android {
    compileSdkVersion 24
    buildToolsVersion "24.0.2"
    defaultConfig {
        ...
    }

    buildTypes {
        release {
            minifyEnabled true
            ...
        }
    }
}

dependencies {
  ...
}

Example of a build.gradle file for a library included in the project and not having such a setting:

apply plugin: 'android-library'

dependencies {
  compile 'com.android.support:support-v4:18.0.+'
}

android {
  compileSdkVersion 14
  buildToolsVersion '17.0.0'
  ...
}

And a rule for Checkmarx:

ProGuardObfuscationNotInUse 

// Search for the release method among all methods in Gradle files
CxList releaseMethod = Find_Gradle_Method("release");

// All objects from build.gradle files
CxList gradleBuildObjects = Find_Gradle_Build_Objects();

// Search for what is inside the "release" method among all objects from build.gradle files
CxList methodInvokesUnderRelease = gradleBuildObjects.FindByType(typeof(MethodInvokeExpr)).GetByAncs(releaseMethod);

// Search inside gradle files for the line "com.android.library" - this means that this file belongs to the library and should be excluded from the rule
CxList android_library = gradleBuildObjects.FindByName("com.android.library");

// Initialize an empty array
List<string> libraries_path = new List<string> {};

// Go through all the found "child" files
foreach(CxList library in android_library)
{
    // Get the path to each file
	string file_name_library = library.GetFirstGraph().LinePragma.FileName;
    
    // Add it to our array
	libraries_path.Add(file_name_library);
}

// Search for all calls to enable obfuscation in release settings
CxList minifyEnabled = methodInvokesUnderRelease.FindByShortName("minifyEnabled");

// Get the parameters of these calls
CxList minifyValue = gradleBuildObjects.GetParameters(minifyEnabled, 0);

// Search among them for enabled ones
CxList minifyValueTrue = minifyValue.FindByShortName("true");

// A little magic, if not found in the standard way :D
if (minifyValueTrue.Count == 0) {
	minifyValue = minifyValue.FindByAbstractValue(abstractValue => abstractValue is TrueAbstractValue);
} else {
    // And if found, then leave the previous result
	minifyValue = minifyValueTrue;	
}

// If no such methods are found
if (minifyValue.Count == 0)
{
    // For more correct display of the trigger location in the file, search for either buildTypes or android
	CxList tempResult = All.NewCxList();
	CxList buildTypes = Find_Gradle_Method("buildTypes");
	if (buildTypes.Count > 0) {
		tempResult = buildTypes;
	} else {
		tempResult = Find_Gradle_Method("android");
	}
	
	// For each of the found trigger locations, determine whether it is a child or main build file
	foreach(CxList res in tempResult)
	{
        // Determine in which file the buildType or android methods were found
		string file_name_result = res.GetFirstGraph().LinePragma.FileName;
        
        // If such a file is not in our list of "child" files, then it is the main file and can be added to the result
		if (libraries_path.Contains(file_name_result) == false){
			result.Add(res);
		}
	}
}

This approach can be quite universal and useful not only for Android applications but also for other cases where it is necessary to determine the belonging of the result to a specific file.

Task: Add support for a third-party library if the syntax is not fully supported

Solution: The number of various frameworks used in the coding process is simply off the charts. Of course, Checkmarx does not always know about their existence, and our task is to teach it to understand that certain methods belong to this framework. Sometimes this is complicated by the fact that frameworks use function names that are very common and it is impossible to unequivocally determine the relationship of a particular call to a specific library.

The difficulty is that the syntax of such libraries is not always correctly recognized and you have to experiment to avoid a large number of false positives. There are several options to improve the accuracy of scanning and solve the task:

  • The first option, we know for sure that the library is used in a specific project and can apply the rule at the team level. But if the team decides to use a different approach or uses several libraries with overlapping function names, we can get an unpleasant picture of numerous false positives.

  • The second option is to search for files where the library is explicitly imported. With this approach, we can be sure that the library is used in this file.

  • And the third option is to use both of the above approaches together.

As an example, let's look at the well-known in narrow circles library slick for the Scala programming language, namely, the functionality of Splicing Literal Values. In general, to pass parameters in an SQL query, you need to use the $ operator, which substitutes data into a pre-formed SQL query. That is, it is essentially a direct analogue of Prepared Statement in Java. But, if you need to dynamically construct an SQL query, for example, if you need to pass table names, you can use the #$ operator, which directly substitutes data into the query (almost like string concatenation).

Example code:

// In general, user-controlled values
val table = "coffees"
sql"select * from #$table where name = $name".as[Coffee].headOption

Checkmarx does not yet know how to determine the use of Splicing Literal Values and skips the #$ operators, so let's try to teach it to identify potential SQL injections and highlight the necessary places in the code: 

// Find all imports
CxList imports = All.FindByType(typeof(Import));

// Search by name, is there slick in the imports
CxList slick = imports.FindByShortName("slick");

// Some flag indicating that the library import is present in the code
// For more accurate determination, you can apply the approach with the file name
bool not_empty_list = false;
foreach (CxList r in slick)
{
    // If we meet an import, we consider that slick is used
	not_empty_list = true;
}

if (not_empty_list) {
    // Search for calls to which an SQL string is passed
	CxList sql = All.FindByShortName("sql");
	sql.Add(All.FindByShortName("sqlu"));
	
	// Determine the data that gets into these calls
	CxList data_sql = All.DataInfluencingOn(sql);
	
	// Since the syntax is not supported, you can apply the approach with regular expressions
	// RegExp should be used very carefully and not applied to a large amount of data, as this can greatly affect performance
	CxList find_possible_inj = data_sql.FindByRegex(@"\#\$", true, true, true);

    // Get rid of unnecessary triggers, if any, and output to the result
	result = find_possible_inj.FindByType(typeof(BinaryExpr));
}

Task: Search for used vulnerable functions in Open-Source libraries

Solution: In many companies, tools are used to control Open-Source (OSA practice), allowing the detection of the use of vulnerable versions of libraries in developed applications. Sometimes it is not possible to update such a library to a safe version. In some cases, there are functional limitations, in others, there is no safe version at all. In such a case, a combination of SAST and OSA practices will help, allowing you to determine that the functions that lead to the exploitation of the vulnerability are not used in the code.

But sometimes, especially when considering JavaScript, this may not be a trivial task. Below is a solution, perhaps not ideal, but nevertheless working, using the example of vulnerabilities in the lodash component in the template and *set methods.

Examples of test potentially vulnerable code in a JS file:

/**
 * Template example
 */

// Find all strings: containing the string lodash (assuming this is the library import declaration
CxList lodash_strings = Find_String_Literal().FindByShortName("*lodash*");

// Find all data: interacting with these strings
CxList data_on_lodash = All.InfluencedBy(lodash_strings);


// Set the list of vulnerable methods
List<string> vulnerable_methods = new List<string> {"template", "*set"};

// Find all our vulnerable methods listed in the vulnerabilities and filter them only where they were called
CxList vulnerableMethods = All.FindByShortNames(vulnerable_methods).FindByType(typeof(MethodInvokeExpr));

//Find all data: interacting with these methods
CxList vulnFlow = All.InfluencedBy(vulnerableMethods);

// If there is an intersection of these data, put it in the result
result = vulnFlow * data_on_lodash;

// Form a list of paths we have already passed to filter duplicates in the future
List<string> lodash_result_path = new List<string> {};

foreach(CxList lodash_result in result)
{
    // Once again, get the paths to the files
	string file_name = lodash_result.GetFirstGraph().LinePragma.FileName;
	lodash_result_path.Add(file_name);
}

// Next is the part related to html files, as we cannot trace where exactly the call is coming from
// Form an array of file paths to be sure that the vulnerable method triggers were in the same files where lodash is declared
List<string> lodash_path = new List<string> {};
foreach(CxList string_lodash in lodash_strings)
{
	string file_name = string_lodash.GetFirstGraph().LinePragma.FileName;
	lodash_path.Add(file_name);
}

// Iterate through all vulnerable methods and make sure they are called in the same files where lodash is declared/included
foreach(CxList method in vulnerableMethods)
{
	string file_name_method = method.GetFirstGraph().LinePragma.FileName;
	if (lodash_path.Contains(file_name_method) == true && lodash_result_path.Contains(file_name_method) == false){
		result.Add(method);
	}
}

// Remove all UknownReferences and leave the "longest" of the paths, if any
result = result.ReduceFlow(CxList.ReduceFlowType.ReduceSmallFlow) - result.FindByType(typeof(UnknownReference));

Task: Search for certificates embedded in the application

Solution: It is not uncommon for applications, especially mobile ones, to use certificates or keys to access various servers or check SSL-Pinning. From a security point of view, storing such things in the code is not the best practice. Let's try to write a rule that will search for such files in the repository: 

// Find all certificates by file mask
CxList find_certs = All.FindByShortNames(new List<string> {"*.der", "*.cer", "*.pem", "*.key"}, false);

// Check where they are used in the application
CxList data_used_certs = All.DataInfluencedBy(find_certs);

// And for mobile applications, we can search for methods where certificate reading is called
// For other platforms and applications, there may be different methods
CxList methods = All.FindByMemberAccess("*.getAssets");

// The intersection of sets will give us the result of using local certificates in the application
result = methods * data_used_certs;

Task: Search for compromised tokens in the application

Solution: It is not uncommon to have to revoke compromised tokens or other important information present in the code. Of course, storing them inside the source code is not the best idea, but situations vary. Thanks to CxQL queries, finding such things is quite simple: 

// Get all strings contained in the code
CxList strings = base.Find_Strings();

// Search among all strings for the value we need. In the example, a token in the form of the string "qwerty12345"
result = strings.FindByShortName("qwerty12345");
