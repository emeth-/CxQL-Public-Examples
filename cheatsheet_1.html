Hello, Habr!

In this article, I want to share our experience of creating our own queries in Checkmarx SAST.

At first glance, it may seem that apart from finding weak encryption/hashing algorithms and a bunch of false positives, it doesn't provide much else. But with proper configuration, it is a super powerful tool that can find serious bugs.

We will delve into the intricacies of the Checkmarx SAST query language and write 2 queries to find SQL injections and Insecure Direct Object References.


Introduction


After a long search for any guides or articles on Checkmarx, it became clear to me that apart from the official documentation, there is little useful information. And even in the official documentation, it's not very clear and understandable. For example, I couldn't find any best practices on how to properly organize override queries, how to write queries "for dummies", etc. Yes, there is documentation on CMx Query Language functions, but how to combine these functions into a single query is not written in the documentation.

Perhaps the lack of articles and guides from the Checkmarx community is due to the high cost of the tool and, as a result, a small audience. Or maybe few people bother with fine-tuning and use the solution as is, out of the box.

In my experience, I see more that SAST is used rather to comply with formalities related to various customer requirements than to find real bugs. With this approach, we end up with, at best, a relatively small number of "vulnerabilities" that are almost automatically clicked as "not exploitable" (because they are in 99.9% of cases).

It should be noted that Checkmarx itself tries to update its queries to give the best result "out of the box". But CMx Query Language queries are tailored for the "general case". The primary token search is based on the name. For example, CMx SAST assumes that all database queries will look like this: *createQuery* or *createSQLQuery*. But if internal developments are used to work with the database, and the method for querying the database is called differently, for example, *driveMyQuery*, then all SQL methods will be missed. For example, our customer uses a custom ORM for SQL DB. In this case, CMx queries "out of the box" missed all SQL injections.

CMxQL – Checkmarx SAST query language
Token – a string with a specific meaning, the result of the lexical analyzer (also called tokenization)

CMx: General Information and Basic Definitions


Before Starting Query Development


Query development is done in a separate program called CxAuditor. In CxAuditor, you need to scan all the code (create a local project) for which we will write queries. After that, you can write and run new queries. For a large codebase, the initial scan can take hours and gigabytes of memory. After that, each query will run relatively quickly. This is not suitable for development.

Therefore, you can take a small set of files from the project, ideally with a pre-found bug of the type for which we are writing the query (or plant a bug there manually) and scan only this set of files. It is not necessary to follow the project's file structure. That is, if you have Java packages A and B, and classes in package B use classes and methods from package A, you can dump it all into one directory, and CMx will still understand the relationships and build call chains between files correctly (or almost always correctly, although errors are unlikely to be related to the project's file structure).

Basic Definitions


CxList


The main data type in CMx. The result of almost all CMxQL functions will be CxList. This is a set of elements with certain properties. The most useful properties for development will be considered further.

result


CMxQL has a built-in variable result. The set contained in the result variable, after the entire query is executed, will be output as the result.

That is, the final operation of any query should be the line result=WHATEVER, for example: 
result = All.FindByName("anyname");

flow and code element


Most CMxQL functions by return type are divided into 2 types: those that return "code elements" and those that return Flow. In both cases, the result will be CxList. But its content will differ slightly for "Flow" and "code elements".
  • Code element — token — for example, a variable, method call, assignment, etc.;
  • Flow — the relationship between the specified tokens.


All and "sub" All


Each CMxQL function can be executed either on the set All (contains all tokens of the entire scanned code, we have already seen an example with result) or on the set CxList, which in turn was obtained as a result of some operations in the query, for example, the query:
CxList newList = CxList.New();

will create an empty set, which we can then fill with elements using the Add() method, and then perform a search on the elements of the new set:
CxList newFind = newList.FindByName("narrowedScope");

Properties of Found Elements


Each element of the CxList set has several properties. When analyzing results for writing queries, the most useful are:

  • SourceFile — the name of the file containing this element;
  • Source Line — the line number with the token;
  • Source Name — the name of the token. Equivalent to the token, so if the variable is called var1, then Source Name = var1;
  • Source Type — the type of the token. For example, if it is a string, it will be StringLiteral, if it is a method call, it will be MethodInvokeExpr, and many others;
  • Destination File
  • Destination Line;
  • Destination Name;
  • Destination Type.


Source and Destination will be different if the elements of the resulting set are Flow, and will coincide if the result is code elements.

Starting to Create Queries


All CMxQL functions can be divided into several types. Here, in my opinion, is the main drawback of the CMxQL documentation: all functions in the documentation are described simply in alphabetical order, whereas it would be much more convenient to structure them by functionality and then by alphabet.

  • Search functions — almost all CMxQL functions with the name FindBy* and GetBy*;
  • Set operations functions – addition, subtraction, intersection, iteration over elements, etc.;
  • Analysis functions — mainly functions *InfluencedBy* *InfluencingOn*.


The main principle of queries is the alternation of these types of functions. First, using search functions, we select only the tokens we are interested in by certain properties. Using set operations, we can combine different sets with different token properties into one, or vice versa, subtract one from another. Then, using analysis functions, we build the Code Flow and try to understand whether potentially vulnerable places depend on parameters at entry points.

The choice of where to start searching and the entire search path depends on the specific code, or rather, even on the "text". In some cases, it is convenient to search from the user request entry point, in others it is more convenient to start from the "end" or even from the middle. It all depends heavily on the specific code and requires an individual approach to each repository.

Example: Finding SQL Injections


Search plan, in parentheses I indicated the names of the sets (variables in the query):

  1. Define exceptions – tokens that can be immediately discarded from the search scope (exclusionList);
  2. Define sanitization/security check places (sanitization);
  3. Find all low-level places with database query execution (runSuperSecureSQLQuery);
  4. Find all parameters of the called methods runSuperSecureSQLQuery (runSSSQParams);
  5. Find entry points (parent methods and their parameters) for database query execution places (entryPointsParameters );
  6. Find dependencies of runSSSQParams parameters on entryPoints , only where there is no input sanitization sanitization.


As a result, we will get low-level methods with SQL queries where the SQL query parameters:

  • depend on method parameters;
  • parameters are accepted as strings;
  • parameters are concatenated to the query.

We will not check whether we can control these parameters, as we assume that there is a mechanism for mapping variables to the query and type conversion for numbers, and string concatenation is always considered dangerous. Even if there is no control over the string now, it may appear in a new release.

SQLi: Step 1. Define Exceptions


Exceptions need to include those classes or files where token names may match the ones we are looking for, as these tokens will lead to invalid findings.

For example, the method for database access is called runSuperSecureSQLquery. We assume that the runSuperSecureSQLquery method is implemented securely inside. Our task is to find places where the method itself is used insecurely. For SQL injection, insecure places will be where user-controlled parameters are concatenated. And secure places will be where parameters are mapped to an ORM structure or, for example, for numeric parameters, they are converted to the corresponding type. We don't need to scan all the code that lies "deeper" than runSuperSecureSQLquery, so it's better to exclude it to avoid useless findings.

To find such exceptions, it is convenient to use CMxQL functions:
  • FindByFileName() – will find a set of all tokens in a specific file;
  • GetByClass()– will find a set of all tokens in a class with a given name.


For the test application, such an exception is the Session class, which contains the implementation of the runSuperSecureSQLquery method.
Example query to exclude code in the Session class (the GetByClass() method will check which of the tokens passed to it has the CMx type ClassDecl, and will return a set of tokens of this class)

CxList exclusionList = All.GetByClass(All.FindByName("*Session*"));
result = exclusionList;


Or another way — exclude code in the entire Session.java file:

CxList exclusionList = All.FindByFileName("*Session.java");
result = exclusionList;

The asterisk before the name is important, as the file name includes the entire path.
Now we have a set of tokens that can be subtracted in the following steps from the search scope.

SQLi: Step 2. Define Sanitization Places


In the test application, there are 2 API methods (see the brief description of the test application). The difference between the two API methods is that getTransactionInfo() concatenates the transactionId parameter to the SQL query, while getTransactionInfoSecured() first converts transactionId to Long type and then passes it as a string. The vulnerability (parameter concatenation) is embedded in both methods. But thanks to the conversion to Long in getTransactionInfoSecured(), the latter method is not vulnerable to injection, as an attempt to pass an injection (string) will result in a Java Exception.

In this example, we will consider the conversion to Long as a sanitization place. To find such tokens:

CxList sanitization = All.FindByName("*Long*");
result = sanitization;


Example result:



The result includes tokens with the Java type Long and methods getValueAsLong, which internally convert the value to Long type. The result needs to be carefully reviewed to ensure nothing unnecessary is included.

SQLi: Step 3. Find All Low-Level Places with Database Query Execution


The following query will find all places using the runSuperSecureSQLQuery token (which is used for database access):

result = All.FindByName("*runSuperSecureSQLQuery*")

The result of searching by the runSuperSecureSQLQuery token name:


For places where this method is called (the Billing class), only method call tokens (type MethodInvokeExpr) will be found, and for the method declaration place (the Session class), all tokens – variables – will also be found.

We will filter only method call tokens:

CxList runSuperSecureSQLQuery = All.FindByName("*runSuperSecureSQLQuery*").FindByType(typeof(MethodInvokeExpr));
result = runSuperSecureSQLQuery;

Result:


The result includes 7 places, 4 of which are the desired calls to the runSuperSecureSQLQuery() method (the Billing and User classes). 2 are calls to the internal runSuperSecureSQLQuery() method within the Session class, and one more is the add method, which is rather a peculiarity of the CMxQL search. Let's say I didn't expect it to be in the list =) The tokens in the Session class, as we found out in step 1, are not of interest to us, so we will simply subtract them from the result in the next step:

CxList runSuperSecureSQLQuery = All.FindByName("*runSuperSecureSQLQuery*").FindByType(typeof(MethodInvokeExpr));
result = runSuperSecureSQLQuery - exclusionList;

We get a valid list of calls to the desired method:



Note the FindByType() and typeof() functions in the previous query. If we want to search by CMx type, that is, by the CxList property "Source Type", we use typeof(Source Type). If we want to search by the programming language data type, we need to pass the parameter as a string. For example:

result  = All.FindByType("String");

will find all Java tokens with the type String.

To do this, first get the names of the parent methods that contain database query calls runSuperSecureSQLQuery, and then get their parameters. To find parent tokens, use the CMxQL function GetAncOfType():

In this query, for the runSuperSecureSQLQuery set, all parent tokens with the type MethodDecl will be returned — this is the previous method in the call stack:

SQLi: Step 6. Find Dependencies of runSSSQParams on entryPointsParameters, Only Where There is No Input Sanitization sanitization


In this step, we use analysis functions. For code flow analysis, the following functions are used:

  • InfluencedBy()
  • InfluencedByAndNotSanitized()
  • InfluencingOn()
  • InfluencingOnAndNotSanitized()
  • NotInfluencedBy()
  • NotInfluencingOn()


To find flow dependencies of the runSSSQParams query parameters on the entryPointsParameters and exclude sanitization tokens:

CxList dataInflOnTable = runSSSQParams.InfluencedByAndNotSanitized(entryPointsParameters, sanitization);


At the same time, I am not sure that the *AndNotSanitized functions do some magic inside, and it looks more like the method simply subtracts the sanitized set from its result. That is, if you do:

CxList dataInflOnTable = runSSSQParams.InfluencedBy(entryPointsParameters) - sanitization;


it turns out the same. Although maybe I just didn't find a case where there are still differences.

To select a specific flow, use the ReduceFlow(CxList.ReduceFlowType flowType) method, where flowType can be:

  • CxList.ReduceFlowType.ReduceSmallFlow — select the longest flows


SQLi: Final Query for Finding SQL Injections


// 1. Define exceptions
CxList exclusionList = All.GetByClass(All.FindByName("*Session*"));
// 2. Define sanitization places
CxList sanitization = All.FindByName("*Long*");

// 3. Find calls to runSuperSecureSQLQuery()
CxList runSuperSecureSQLQuery = All.FindByName("*runSuperSecureSQLQuery*").FindByType(typeof(MethodInvokeExpr));
runSuperSecureSQLQuery -= exclusionList;

// 4. Find parameters of the called method runSuperSecureSQLQuery()
CxList runSSSQParams = All.GetParameters(runSuperSecureSQLQuery);

// 5. Find parameters of methods that are parents to runSuperSecureSQLQuery()	
CxList entryPoints = runSuperSecureSQLQuery.GetAncOfType(typeof(MethodDecl));
CxList entryPointsParameters = All.GetParameters(entryPoints).FindByType("String");

// 6. Find dependencies of the database query parameters (runSuperSecureSQLQuery) on the parent method parameters
CxList dataInflOnTable = runSSSQParams.InfluencedByAndNotSanitized(entryPointsParameters, sanitization);

// 7. Output the result
result = dataInflOnTable.ReduceFlow(CxList.ReduceFlowType.ReduceBigFlow);


Example 2: Finding Insecure Direct Object References


In this query, we will look for all places where objects are accessed without checking the object's owner. At the same time, different names for the HTTP parameter mailboxid may be used (assuming this is legacy), and the check itself may occur at different stages: somewhere right at the HTTP API entry point, somewhere before the database query, and sometimes in intermediate methods.

Search plan
  1. Define exceptions (exclusionList);
  2. Define authorization check places (idorSanitizer);
  3. Find entry points – places of initial HTTP request processing (webRemoteMethods);
  4. Find places of HTTP parameter mailboxid extraction only by entry point tokens (mailboxidInit);
  5. Find all calls from webRemoteMethods to middleware methods and their parameters (middlewareMethods);
  6. Find middleware methods that depend on mailboxid (apiPotentialIDOR);
  7. Find all places of middleware method definitions (middlewareDecl);
  8. Go through all apiPotentialIDOR and select only those middlewareDecl that do not check the object's owner mailboxid.


IDOR: Step 1. Define Exceptions


In this case, exclude all tokens in a specific file:

CxList exclusionList = All.FindByFileName("*WebMethodContext.java");
     result = exclusionList;

WebMethodContext.java contains the implementation of methods such as getMailboxId and getUserId, as well as the string "mailboxid". Since the token names will match the ones we need to find the vulnerability, this file will give false positives.

IDOR: Step 2. Define Authorization Check Places


In the test application, the method validateMailbox() is used to determine if the requested object belongs to the user:

CxList idorSanitizer = All.FindByName("*validateMailbox*");
     result = idorSanitizer;

HTTP request handlers have a special annotation that makes them easy to find. In my case, it is "WebRemote", to search for annotations, use the CMxQL function FindByCustomAttribute(). For FindByCustomAttribute(), the parent token search function GetAncOfType() will return the method under the annotation:

      .GetAncOfType(typeof(MethodDecl));
result = webRemoteMethods;


Result of the query:



IDOR: Step 4. Find Places of HTTP Parameter mailboxid Extraction Only by Entry Point Tokens


To find tokens related to processing the HTTP parameter mailboxid:

CxList getMailboxId = All.FindByName("\"mailboxId\"") 
+ All.FindByName("\"mid\"") 
+ All.FindByName("\"boxid\"");
result = getMailboxId;

we added 3 sets with 3 different strings, as in the legend, the name of the HTTP parameter may differ in different parts of the system.

The query will find all places where mailboxid/mid/boxid is written as a string (in double quotes). But this query will return a lot of findings, as such a string can appear not only in places where HTTP parameters are extracted. If we continue to work with this set, we will get a huge number of false positives.

Therefore, we will search only by entry point tokens (webRemoteMethods). To find all child tokens, use the CMxQL function GetByAncs():

result = All.GetByAncs(webRemoteMethods);

The query will return all tokens belonging to methods annotated as WebRemote. At this stage, we can already filter the tokens of those methods that perform an owner check. Therefore, we will rewrite the previous query to search for child tokens so that we select only the child tokens of WebRemote methods where there is no security check for the object's owner. To do this, use a loop with a condition:

// create an empty set for child tokens of potentially dangerous methods
CxList entry_point_tokens = All.NewCxList();
// iterate over the set of parent tokens webRemoteMethods
foreach (CxList method in webRemoteMethods) {
	// get all child tokens of the current parent token
	CxList method_tokens = All.GetByAncs(method);
	// check if the list of child tokens contains tokens used for owner check
	if (method_tokens.FindByName(idorSanitizer).Count > 0) {
	// if yes, do nothing, consider this method safe
	} else { 
	// if no, add the method's tokens to the list of potentially dangerous ones
		entry_point_tokens.Add(method_tokens);
	}
}

Now we can make a more accurate selection by HTTP parameters mailboxid:

CxList getMailboxHTTPParams = entry_point_tokens.FindByName("\"mailboxid\"") 
+ entry_point_tokens.FindByName("\"mid\"") 
+ entry_point_tokens.FindByName("\"boxid\"");
result = getMailboxHTTPParams;

But we are not interested in the places where HTTP parameters are extracted, but in the variables to which the values of HTTP parameters are ultimately assigned. Since it is more reliable to search for flow by variable tokens.

The CMxQL function FindByInitialization() will find the places where variables are initialized for the given tokens:


CxList mailboxidInit = entry_point_tokens.FindByInitialization(getMailboxHTTPParams);
result = mailboxidInit;

IDOR: Step 5. Find All Calls from webRemoteMethods to Middleware Methods and Their Parameters


By middleware, I mean code that goes deeper than HTTP API request handling methods, that is, deeper than user request entry points. For example, for the screenshot above, these are the methods of the User class, calls to user.getSettings() and user.getSecureSettings():

CxList middlewareMethods = All.FindByShortName("user").GetRightmostMember();
CxList middlewareMethodsParams = entry_point_tokens.GetParameters(middlewareMethods);
result = middlewareMethodsParams;

First, select all tokens with the name user, and then use GetRightmostMember() to select tokens of calls to middleware. GetRightmostMember() in the method call chain will return the rightmost one. After that, output the parameters of the found method using GetParameters().

Result:



IDOR: Step 6. Find Middleware Methods That Depend on mailboxid


For flow analysis, use the *InfluencedBy* and *InfluencingOn* methods. The difference between them is clear from the name.

For example:

All.InfluencedBy(getMailboxHTTPParams)

will go through the All set and find all tokens that depend on getMailboxHTTPParams.

The same can be written in another way:

getMailboxHTTPParams.InfluencingOn(All)


To find tokens that depend on mailboxidInit:

CxList apiPotentialIDOR = entry_point_tokens.InfluencedByAndNotSanitized(mailboxidInit, idorSanitizer);
result = apiPotentialIDOR;

Result:



IDOR: Step 7. Find All Places of Middleware Method Definitions


Find the definitions of all intermediate methods that can be used in user request handling places. To do this, highlight their common property, for example, all such methods have the creation of a Request() object, the creation of an object has the CMx type ObjectCreateExpr:

CxList requests = (All - exclusionList).FindByType(typeof(ObjectCreateExpr)).FindByName("*Request*");
CxList middlewareDecl = requests.GetAncOfType(typeof(MethodDecl));
result = middlewareDecl;


(All — exclusionList) — you can make such a set subtraction, after which call the desired CMxQL function from the resulting result. Now requests contains all tokens with the name Request and the type corresponding to object creation.

Then, using the already familiar GetAncOfType(), find the parent token with the type MethodDecl.

Result:



IDOR: Step 8. Go Through All apiPotentialIDOR and Select Only Those middlewareDecl That Do Not Check the Object's Owner mailboxid


In the final part of the query, we will determine which of the middleware methods are called directly from the entry point methods and do not check who owns the mailboxid. After that, we will combine the flow for more convenient analysis of the results.

New functions that we have not used yet:
GetCxListByPath() – this function is needed for iterating over the flow, if you do NOT use it, CMx will compress the flow into a Code Element (into the first node of the flow)
Concatenate*() — a series of functions needed to combine several flows into one
FindByParameters() – find a method by a specific token parameter
GetName() – will return a string with the token name, if there is more than one element in the CxList, it will return the first one. The method is used only when iterating over elements of the set.

The final part of the query:

// create an empty set
CxList vulns = All.NewCxList();
// iterate over the flow set apiPotentialIDOR
foreach(CxList cxFlow in apiPotentialIDOR.GetCxListByPath()) {
  // extract the last node of the flow
  CxList endNode = cxFlow.GetStartAndEndNodes(CxList.GetStartEndNodesType.EndNodesOnly);
  // find the method call by the parameter from the flow (mailboxid)
  CxList method_call = entry_point_tokens.FindByParameters(endNode);
  // find the definition of this method
  CxList method_decl = middlewareDecl.FindByShortName(method_call.GetName());

  // if the method definition is found
  if (method_decl.Count > 0) {	
    // extract all tokens within this method
    CxList _all = (All - exclusionList).GetByAncs(method_decl);
    // check if the method has sanitization
    if (_all.FindByName(idorSanitizer).Count > 0) {
      // if yes, just add a record to the log
      cxLog.WriteDebugMessage("find sanitized in method: " + method_call.GetName());
    // if no, add the flow to the set of potentially dangerous vulns
    } else {
      // at the same time, add the method call and its definition tokens to the flow
      vulns.Add(cxFlow.ConcatenatePath(method_call).ConcatenatePath(method_decl));
      cxLog.WriteDebugMessage("find NOT sanitized in method: " + method_call.GetName());
    }
  }
}


Result:



CocatenatePath we used to make it convenient to navigate through the code when analyzing all findings. This method attaches one Code Element to the flow

if (method_tokens.FindByName(idorSanitizer).Count > 0) {
} else { 
  entry_point_tokens.Add(method_tokens);
}
}
// Search for places of HTTP parameter extraction and initialization of the corresponding variable

    }

Conclusion


Checkmarx parses the code into tokens without any problems, while determining their types. The static analyzer also performs simple token searches well, for example, finding the parent token for the current one, finding variable initialization, finding method parameters, etc. It almost as well builds flows (but sometimes it still misses). All this allows you to work with the code as with any database, with the difference that the code structure is not predefined, and you have to "adjust" it yourself.

To significantly reduce the number of false positives, pay attention to the following:
  • Adding exceptions in project settings to exclude unnecessary files and directories (e.g., directories with tests).
  • Review the out-of-the-box queries and disable unnecessary ones (or modify the query). For example, there is a bug type "Privacy Violation", it looks for places where sensitive data can be either written to a file or sent to the Web UI. The latter is not particularly useful, as most often displaying sensitive data in the UI is part of the business logic. In this case, protection is provided by TLS to protect the channel and an audit for XSS to protect data in the browser.
  • Sometimes a bug may not be found because the out-of-the-box query did not include the default token (yes, this is certainly a flaw). For example, to search for XXE, check if all tokens corresponding to the default library names are in the query.
  • If there are many false positives that are completely off, you need to look at the out-of-the-box query and compare the token names in the CMxQL functions of the FindBy/GetBy type. Most likely, the names of the expected tokens will differ from those used in the code (at the very beginning, I gave an example with missed SQL injections).
  • If there are many false positives that found the right places, but there is no bug in the end, then you are lucky, most likely you need to slightly adjust the query to indicate to CMx what is considered sanitization for this query. For example, we found the right places with LDAP queries where the query fields are partially controlled by users. But in fact, each method with an LDAP query had another method that performed sanitization, and Checkmarx did not understand this.


I hope this how-to will be useful as a "hello world" for those who are just starting to write their own queries in Checkmarx.
